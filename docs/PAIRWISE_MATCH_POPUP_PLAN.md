# Pairwise Match Popup Implementation Plan

## Objective

Add hover and click popups for linear pairwise match ribbons/curves, including plain genome matches, orthogroup-derived matches, and collinearity block matches.

The implementation should keep feature popups and match popups conceptually separate:

- Feature popup: details about one genomic feature.
- Match popup: details about one relationship between two regions/features.

This separation keeps the model simple, prevents feature-editing controls from appearing on match objects, and avoids coupling pairwise match behavior to unrelated feature editor actions.

## Design Principles

- SOLID: keep SVG metadata emission, in-app interaction, and standalone interactive SVG behavior in separate modules with clear responsibilities.
- KISS: use SVG `data-*` attributes already emitted on pairwise match paths as the primary contract.
- YAGNI: do not add match search, editing, filtering, or persistence until users need those workflows.
- DRY: reuse existing popup positioning, hover delay, copy buttons, highlight helpers, and standalone SVG overlay code where possible.

## Current State

- Pairwise match shapes are generated by `gbdraw/render/groups/linear/pairwise_match.py`.
- Some match metadata is already emitted, including:
  - `data-pairwise-match-style`
  - `data-identity-factor`
  - `data-orthogroup-id`
  - `data-collinearity-block-id`
  - query/subject protein and feature SVG IDs for LOSATP/collinear modes
- Web preview feature interactions live in `gbdraw/web/js/app/feature-editor/svg-actions.js`.
- Standalone interactive SVG export is assembled in `gbdraw/web/js/services/standalone-interactivity.js`.
- Standalone SVG already uses `[data-orthogroup-id]` pairwise paths for orthogroup hover highlighting, but those paths are not clickable popup targets.

## Target Behavior

### Hover

On desktop pointer devices, hovering over a pairwise match should:

- highlight the hovered match;
- for orthogroup matches, also highlight linked member features and same-orthogroup match elements;
- for collinearity matches, highlight same block elements;
- show a compact delayed summary popup.

The hover popup should not appear while:

- a click popup is open;
- the user is panning/dragging;
- the pointer is over viewport controls or search controls.

### Click

Clicking a pairwise match should open a fixed, draggable detail popup.

Feature elements remain feature targets. If a click target could be interpreted as both feature and match, feature should win only for actual feature shapes. Pairwise match paths should open match popups.

### Popup Content

Plain pairwise match:

- title: `Pairwise match`
- query record and subject record
- query interval and subject interval
- identity, alignment length, e-value, bit score
- mismatches and gap opens when available
- match style and orientation when available

Orthogroup match:

- title: `Orthogroup match`
- orthogroup ID, display name, description
- member count and record coverage
- query feature: record, location, protein ID/source protein ID, gene/locus/product when available
- subject feature: same fields
- alignment statistics

Collinearity block match:

- title: `Collinearity block`
- block ID, kind, orientation, color mode
- block score/e-value/anchor count when available
- selected anchor details: query unit, subject unit, display names, feature/protein IDs
- alignment statistics

## Data Contract

### Required attributes on each pairwise match path

Add stable metadata in `PairWiseMatchGroup.generate_linear_match_path()` and `add_optional_metadata_attributes()`:

- `data-gbdraw-pairwise-match-id`
- `data-match-kind`
- `data-query-record-index`
- `data-subject-record-index`
- `data-query-record-id`
- `data-subject-record-id`
- `data-qstart`
- `data-qend`
- `data-sstart`
- `data-send`
- `data-identity`
- `data-alignment-length`
- `data-evalue`
- `data-bitscore`
- `data-mismatches`
- `data-gap-opens`

`data-match-kind` values:

- `pairwise`
- `orthogroup`
- `collinear`

Derivation:

- `collinear` when `collinearity_block_id` is present.
- `orthogroup` when `orthogroup_id` is present without `collinearity_block_id`.
- `pairwise` otherwise.

### Optional attributes

Keep and extend current optional attributes:

- `data-collinearity-block-id`
- `data-collinearity-block-kind`
- `data-collinearity-orientation`
- `data-collinearity-block-score`
- `data-collinearity-block-evalue`
- `data-collinearity-anchor-index`
- `data-collinearity-anchor-count`
- `data-collinearity-color-mode`
- `data-orthogroup-id`
- `data-query-protein-id`
- `data-subject-protein-id`
- `data-query-feature-svg-id`
- `data-subject-feature-svg-id`
- `data-query-unit-id`
- `data-subject-unit-id`
- `data-query-locus-id`
- `data-subject-locus-id`
- `data-query-display-name`
- `data-subject-display-name`

Do not embed full sequences or full qualifier maps in pairwise match path attributes.

## Implementation Phases

### Phase 1: SVG metadata emission

Files:

- `gbdraw/render/groups/linear/pairwise_match.py`
- targeted SVG tests under `tests/`

Tasks:

1. Add helpers for safe string/number attribute emission.
2. Assign deterministic `data-gbdraw-pairwise-match-id` values within each comparison group.
3. Emit the required match attributes for all modes.
4. Extend optional collinearity attributes to include score, anchor index/count, locus IDs, and display names when present.
5. Keep generated static SVG visually unchanged.

Acceptance:

- Existing visual reference output changes should be limited to added metadata attributes.
- Pairwise, orthogroup, and collinear examples expose enough attributes for popup rendering without parsing path geometry.

### Phase 2: Web preview match interactions

Files:

- `gbdraw/web/js/app/feature-editor/svg-actions.js`
- optional small helper module under `gbdraw/web/js/app/` if the added code would make `svg-actions.js` too broad

Tasks:

1. Add `PAIRWISE_MATCH_SELECTOR`.
2. Build a lightweight match payload from clicked path attributes.
3. Add delegated hover/click handlers for pairwise match paths.
4. Reuse existing hover positioning logic where practical.
5. Add match-specific highlight helpers:
   - exact match hover;
   - same orthogroup;
   - same collinearity block.
6. Store match popup state separately from `clickedFeature`.

Acceptance:

- Clicking a feature still opens the existing feature popup.
- Clicking a pairwise match opens a match detail popup.
- Hovering feature and match targets does not leave stale highlight classes/styles.

### Phase 3: Web preview popup UI

Files:

- `gbdraw/web/index.html`
- `gbdraw/web/js/state.js`
- `gbdraw/web/js/app/app-setup.js`

Tasks:

1. Add `clickedPairwiseMatch`, position, drag, and optional resize state.
2. Add a compact match popup template, separate from the feature editor popup.
3. Use the same visual language as the feature popup, but omit feature editing controls.
4. Add rows based on `matchKind`, showing only available values.
5. Keep the popup small by default and let it scroll when content is long.

Acceptance:

- Popup text fits on desktop and narrow screens.
- Popup can be closed with the close button and when clicking empty canvas space.
- The feature popup remains unchanged except for any shared helper extraction.

### Phase 4: Standalone interactive SVG support

Files:

- `gbdraw/web/js/services/standalone-interactivity.js`
- `gbdraw/web/js/services/export.js` only if option plumbing is needed

Tasks:

1. Add `MATCH_SELECTOR` inside the standalone script.
2. Build `matches` metadata from rendered SVG pairwise paths during export.
3. Include `matches` in the embedded metadata JSON.
4. Mark pairwise match paths with an interactive class.
5. Add `closestMatch()` and match-specific event handling.
6. Reuse the existing `foreignObject` popup infrastructure.
7. Render feature and match popups through separate render functions sharing row/table/copy utilities.
8. Ensure feature search still targets only features.

Acceptance:

- Downloaded `*.interactive.svg` supports match hover/click without the web app.
- Existing feature popups, viewport controls, search controls, panning, and zooming still work.
- Gallery SVGs can be regenerated later without hand editing generated files.

### Phase 5: Tests and verification

Suggested checks:

- Unit or snapshot-style tests for match `data-*` attributes.
- Web packaging tests for standalone interactivity metadata schema.
- Browser/manual verification with:
  - plain uploaded BLAST/TBLASTX pairwise mode;
  - LOSATP `pairwise`;
  - LOSATP `orthogroup`;
  - LOSATP `collinear`.

Manual verification checklist:

- Feature click still opens feature popup.
- Match click opens match popup.
- Hover summary appears after delay and follows pointer.
- Same-orthogroup and same-collinearity highlights clear on mouseout.
- Exported interactive SVG behaves the same when opened directly from disk.
- Static SVG/PNG/PDF exports remain unaffected.

## Non-Goals

- Editing pairwise match colors from the popup.
- Searching pairwise matches from the feature search box.
- Filtering or hiding individual matches from the popup.
- Embedding full feature qualifiers or sequences in match metadata.
- Reworking pairwise rendering geometry.
- Updating generated gallery SVGs in the same change.

## Risk Management

- Large standalone script: keep new functions small and isolated, and reuse existing popup utilities.
- Dense SVG performance: use delegated event handlers and attribute lookups, not per-path listeners.
- Metadata bloat: store only scalar match fields and references to feature/orthogroup IDs.
- Backward compatibility: if required attributes are missing in older SVGs, render a minimal popup instead of failing.
- Visual regressions: avoid changing path drawing attributes except cursor/class and metadata.

## Proposed Implementation Order

1. Add complete pairwise match path metadata in Python.
2. Add focused tests that assert metadata exists for plain, orthogroup, and collinear rows.
3. Add Web preview match hover/click state and popup.
4. Add standalone interactive SVG match payload and event handling.
5. Run targeted web packaging tests and manually verify one exported interactive SVG.
