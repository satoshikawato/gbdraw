# Interactive SVG Search Performance Implementation Plan

- Created: 2026-07-19
- Target version: `0.14.0b0` or later
- Design snapshot: branch `main`, HEAD `f1182b8`
- Status: implemented and verified on 2026-07-19
- Primary scope: standalone interactive SVG search
- Secondary scope: web app preview search, where the same rendering and indexing problems exist

## 1. Outcome

Make feature search remain responsive for interactive SVGs with approximately 25,000 rendered features while preserving current search semantics and the final highlighted appearance.

The implementation will deliver all five agreed improvements:

1. Disable expensive feature transitions during bulk search-state changes.
2. Replace per-feature dimming mutations with one SVG-root search state and hit-only classes.
3. Update only the old and new active matches during previous/next navigation.
4. Build reusable normalized search indexes instead of reconstructing and lowercasing search values for every query.
5. Compact interactive SVG metadata without removing popup, qualifier, sequence, search, or match functionality.

The work is complete only when standalone exports produced by both the web app and Python paths use the same runtime behavior and equivalent compact metadata.

## 2. Baseline and root cause

A representative linear interactive SVG with 24,945 searchable features produced the following baseline on 2026-07-19:

| Measurement | Baseline |
|---|---:|
| Interactive SVG size | 177,201,707 bytes (169 MiB) |
| Embedded metadata size | 160,256,975 bytes |
| Metadata share of file | 90.4% |
| Feature metadata array | 109,696,957 bytes |
| Match metadata array | 50,559,774 bytes |
| Searchable features | 24,945 |
| SVG feature elements | 24,965 |
| Total DOM elements | 25,871 |

The query `dnaA`, field `Qualifier value`, qualifier key `gene`, returned five features. Search matching took approximately 0.18-0.36 seconds in headless Chromium. The current result application then dimmed 24,960 SVG elements and started an opacity/filter transition on those elements. The renderer remained heavily occupied for tens of seconds.

With feature transitions disabled temporarily, the same query completed its handler in approximately 184 ms, reached two animation frames in 93 ms, and answered a subsequent DOM query in 4 ms. This establishes result rendering, rather than matching alone, as the first bottleneck.

The current implementation has four related costs:

- `applySearchResults()` traverses every rendered feature and toggles three classes for every search.
- The base feature rule transitions `opacity`, `filter`, `stroke`, `stroke-width`, and `stroke-opacity`; a search can start tens of thousands of transitions at once.
- Previous/next navigation calls the same full result application even though only two active elements need to change.
- Search values, sorted qualifier keys, lowercase strings, location text, orthogroup details, and result-detail arrays are rebuilt for each query.

Metadata also contains avoidable duplication. In the representative file, the raw values alone accounted for:

| Field | Value bytes |
|---|---:|
| `nucleotide_sequence` | 23,961,909 |
| `amino_acid_sequence` | 7,814,242 |
| `nucleotide_fasta` | 27,097,638 |
| `amino_acid_fasta` | 10,005,041 |
| `qualifiers.translation` | 7,814,242 |

FASTA text can already be generated by the embedded runtime from sequence and feature metadata. The separate amino-acid sequence is also identical to `qualifiers.translation` for most CDS features in this file.

## 4. Architecture decisions

### 4.1 Preserve the final appearance, not the bulk animation

Matched, active, and unmatched features must retain their current final opacity, stroke, and glow. Bulk search transitions may be skipped because they are transient and cause the browser stall.

Use two root classes:

- `gbdraw-feature-search-active`: indicates that a valid non-empty query is applied.
- `gbdraw-feature-search-updating`: disables feature transitions while the result set is changing or being cleared.

The standalone CSS will dim unmatched features through the root state:

```css
.gbdraw-feature-search-active
  .gbdraw-interactive-feature:not(.gbdraw-interactive-feature--match) {
  opacity: 0.18;
}

.gbdraw-feature-search-updating .gbdraw-interactive-feature {
  transition: none;
}
```

Do not add a `dimmed` class to every unmatched feature. Keep support for removing the legacy class from an already enriched or re-enriched SVG, but stop emitting it in new search updates.

Bulk transition suppression must cover both applying and clearing a query. Use an explicit update scheduler:

1. Add `gbdraw-feature-search-updating`.
2. On the next animation frame, update the root active state and hit classes.
3. On the following animation frame, remove `gbdraw-feature-search-updating`.

This ordering ensures the browser has applied the no-transition rule before opacity changes. Repeated searches must cancel or supersede stale scheduled frames.

The web preview will use equivalent preview-prefixed classes and scheduling. Normal hover behavior outside a search update remains unchanged.

### 4.2 Maintain an applied DOM state

Introduce a small renderer-owned state separate from the logical search state:

```text
appliedSearchDomState
  queryActive: boolean
  matchedIds: Set<string>
  activeId: string
  updateGeneration: number
```

Result application computes a set difference:

```text
removed matches = previous matchedIds - next matchedIds
added matches   = next matchedIds - previous matchedIds
```

Only removed and added match IDs are updated. Each ID may map to multiple SVG parts, so the existing `featureElementsById` map remains the source of DOM elements.

Clearing a search removes classes only from the previous match set and old active match. It does not traverse all rendered features.

### 4.3 Separate result-set updates from active-match updates

Create distinct operations:

- `applySearchResultSet(nextMatches, nextActiveId, queryActive)` updates the root state and match-set difference.
- `applyActiveSearchMatch(previousActiveId, nextActiveId)` updates at most the SVG parts belonging to those two IDs.
- `syncSearchControls()` updates text and buttons without touching diagram feature classes.

`setActiveMatch()` must call only the active-match operation, centering logic, optional popup logic, and control synchronization. It must not call the full result-set operation.

The same separation applies to `createPreviewFeatureSearch()` and `preview-svg.js`.

### 4.4 Use a prepared search index, not a heavyweight search engine

The data set has tens of thousands, not millions, of records. Avoid a general inverted-index dependency and avoid a Web Worker initially because copying or transferring a large rich payload can increase memory pressure.

Build a prepared index once after metadata parsing in a standalone SVG, and rebuild it only when `extractedFeatures` or orthogroups change in the web preview.

The index contract will contain:

```text
FeatureSearchIndex
  featureOrder: string[]
  byId: Map<string, PreparedFeatureSearchDocument>
  qualifierFeatureIdsByKey: Map<string, string[]>
  orthogroupById: Map<string, PreparedOrthogroup>
  orthogroupMemberByFeatureId: Map<string, PreparedMember>
```

Each prepared feature document contains field-specific items with both original and normalized text:

```text
PreparedSearchItem
  label: string
  value: string
  normalizedValue: string
  alphabet: "" | "nucleotide" | "amino-acid"
```

Requirements:

- Build label, type, record, location, strand, orthogroup, qualifier-key, and qualifier-value items once.
- Build lowercase values once for literal text search.
- Index qualifier keys case-insensitively so a qualifier-value search scans only documents that contain the selected key.
- Precompute orthogroup/member lookup maps; do not call `Array.find()` for every feature and query.
- Compile regex and IUPAC patterns once per query.
- Preserve feature order in results.
- For a literal search, stop testing a feature after the first hit unless additional details are needed.
- Materialize complete match-detail objects only for matched features. Do not allocate `.map().filter()` result arrays for every non-hit.
- Keep original values for case-preserving snippets and regex behavior.
- Preserve current rich/simple popup restrictions and all existing field aliases.

Regex remains a scan over the relevant prepared field because arbitrary regular expressions cannot use the literal index safely. Sequence searches remain scans over the relevant sequence field with the existing IUPAC semantics. These paths still benefit from cached field selection and avoided object/string reconstruction.

Do not add trigram indexing in the first implementation. Add it only if the post-rendering-fix literal-search target is still missed and profiling shows matching, rather than rendering, is dominant.

### 4.5 Write compact metadata version 2 and read versions 1 and 2

Write a new schema identifier, `gbdraw-interactive-feature-popup-v2`. The embedded runtime must accept both the current v1 payload and the compact v2 payload. New exports write v2 only.

Keep `InteractiveSvgContext` and upstream extracted-feature contracts unchanged. Compaction occurs at the final serialization boundary so CLI, Python API, session, and web feature models do not acquire compact-wire-format concerns.

#### Feature payload compaction

- Omit `nucleotide_fasta` and `amino_acid_fasta`; generate both on demand with the existing runtime helpers.
- Omit `amino_acid_sequence` when it is identical to the first non-empty `qualifiers.translation` value.
- Add one runtime resolver for amino-acid sequence that checks the explicit field first and then `qualifiers.translation`.
- Omit empty strings, empty arrays, empty objects, false flags, and null/default numeric values when absence has the same meaning.
- Emit one canonical spelling for each v2 field. Keep alias handling only in the reader/search preparation layer.
- Do not duplicate derived display strings if the runtime can produce them from canonical IDs, location, and qualifier values without changing output.
- Retain raw nucleotide sequence because it is not otherwise available in the standalone file.
- Retain qualifiers in rich mode; qualifier display and qualifier search must remain complete.
- Keep simple popup mode free of rich qualifiers and sequences, as it is now.

#### Match payload compaction

The representative file spends 50.6 MB on match metadata. Collinearity blocks currently expand orthogroup entries into UI-ready objects containing repeated empty fields and derived rows.

- Serialize canonical raw match fields rather than pre-rendered `sections` and `hover_rows` where the runtime can derive those views.
- Store collinearity block orthogroup IDs once as an array; do not store both a semicolon-joined ID string and expanded objects for the same IDs.
- Reference top-level orthogroup metadata by ID when it exists.
- When no top-level orthogroup detail exists, retain the ID only and let the runtime render an ID-only row.
- Omit empty `detail_rows`, `member_rows`, `member_copy_text`, alias fields, and empty display metadata.
- Build popup sections and hover rows lazily when a match is hovered or opened.

#### Compatibility

- The v2 runtime must normalize v1 features and matches to the same internal view used by v2.
- Existing v1 interactive SVG files remain self-contained and unchanged; no migration is required.
- New v2 files embed the matching v2-capable runtime, so they do not depend on installed gbdraw code after export.
- Python and JavaScript exporters must produce semantically equivalent v2 metadata from equivalent contexts.
- Any public test that currently asserts serialized FASTA fields must be changed to assert runtime-generated FASTA output and v1-reader compatibility instead.

### 6.1 Search semantics

Test all fields in rich mode:

- All
- Label
- Feature type
- Record ID
- Location
- Strand
- Orthogroup
- Qualifier key
- Qualifier value with and without a qualifier key
- Nucleotide
- Amino acid

For each applicable field, cover:

- Case-insensitive literal substring
- No match
- Multiple matches in stable feature order
- Regex match
- Invalid regex
- Multiple values for one qualifier
- Missing and empty fields
- Alias fields accepted by v1
- IUPAC ambiguity behavior
- Match detail and snippet text
- Simple popup restrictions

### 6.2 DOM and visual behavior

- Applying a query sets the root active state.
- Unmatched features compute to opacity `0.18` without a dimmed class.
- Matches retain full opacity, match glow, and match stroke.
- The active match retains active glow and active stroke.
- Split feature parts update together.
- Result changes remove stale match classes.
- Clear restores normal appearance.
- Invalid regex does not leave stale dimming.
- Previous/next updates only old/new active elements.
- Hover and click remain functional before, during, and after a search.
- Exported static SVG/PNG/PDF content contains no transient preview search classes.

### 6.3 Metadata compatibility

- v1 feature payload with explicit FASTA.
- v1 feature payload without explicit FASTA.
- v2 feature payload with derived FASTA.
- v2 CDS with amino acid only in `qualifiers.translation`.
- v2 non-CDS without translation.
- Simple and rich popup payloads.
- Pairwise, orthogroup, and collinearity matches.
- Empty optional values omitted from v2.
- Equivalent web and Python serialized semantics.

### 6.4 Performance and size

Use operation counts as stable CI assertions and wall-clock thresholds as broad regression guards.

For a generated 25,000-feature, five-hit case:

- Initial result application must not mutate unmatched feature classes.
- Result application mutations must scale with hit SVG parts, not total feature count.
- Previous/next feature-class mutations must be bounded by the old and new active feature parts.
- Clear mutations must scale with the previous hit count.
- Search plus two animation frames should complete within 1 second in the supported headless Chromium CI environment after warm page load.
- Previous/next should complete within 250 ms in the same environment.

For the representative file:

- The browser must accept input immediately after the result reaches two animation frames.
- The main thread must not remain occupied for multi-second transition processing.
- Metadata must be at least 35% smaller than baseline.
- Record peak memory where browser tooling permits; reject a design that reduces file size by constructing another full duplicate of all sequence strings.

## 9. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Root `:not(.match)` selector still causes a full style recalculation | Remove per-element mutations and transitions first; measure two-frame latency. Consider a match overlay only if profiling still shows style recalculation as dominant. |
| Clearing search accidentally starts mass transitions | Keep the updating class active for a frame before changing the root active state, and remove it only after the following frame. |
| Rapid repeated searches apply stale animation-frame work | Use a monotonically increasing update generation and ignore/cancel stale callbacks. |
| Index duplicates large sequence strings | Store references to original strings and one lowercase copy only where needed; measure index memory. Do not lowercase nucleotide/amino-acid sequences unless the measured matcher benefits. |
| Regex behavior changes under indexing | Run regex over original prepared values and preserve existing flags and result order. |
| V2 breaks older metadata consumers | Version the schema, retain the v1 reader, keep upstream public context unchanged, and document the wire-format change in release notes. |
| Web and Python serializers drift | Use shared semantic fixtures and parity assertions for canonical v2 payloads. The Python runtime continues loading the shared JS runtime/style asset. |
| Match compaction loses popup detail | Compare normalized popup sections, hover rows, copy text, and open-feature links between v1 and v2 fixtures. |
| Absolute timing assertions are flaky | Gate CI primarily on mutation complexity and use generous timing ceilings; keep representative-file measurements as a recorded benchmark. |

## 10. Definition of done

All of the following must be true:

- The five agreed improvements are implemented in standalone interactive SVG search.
- Equivalent rendering/indexing improvements are applied to web preview search where applicable.
- Search semantics and result order are unchanged for all supported fields.
- The final match, active-match, and dimmed appearance is unchanged.
- Bulk searches no longer start transitions on tens of thousands of features.
- Search, clear, and result changes update only the root and changed hit elements.
- Previous/next updates only the old and new active feature elements.
- Search values are prepared once per feature-data revision rather than once per query.
- New exports write compact v2 metadata; the runtime reads v1 and v2.
- FASTA copy/download output remains identical while precomputed FASTA fields are absent from v2.
- The representative metadata is at least 35% smaller.
- The generated 25,000-feature browser case meets the mutation and responsiveness targets.
- CLI, Python API, web export, offline opening, popup, sequence, and match workflows pass focused verification.
- Final benchmark results and any user-visible compatibility notes are documented.
